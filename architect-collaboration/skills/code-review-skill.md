# Code Review Skill - 代码审查技能

本技能提供专业、系统的代码审查能力，适用于 Phase 4（功能开发）完成后的质量把关。

## 审查原则

### 1. 建设性原则
- 指出问题的同时提供改进建议
- 区分问题严重等级（阻塞/严重/建议）
- 关注代码而非个人

### 2. 全面性原则
- 安全性、性能、规范性、可维护性全覆盖
- 结合业务场景评估技术决策
- 考虑长期维护成本

### 3. 务实原则
- 优先解决高风险问题
- 避免过度工程化建议
- 尊重团队技术选型

---

## 审查维度

### 1. 安全性审查 (Security) - 权重 25%

| 检查项 | 说明 | 检查方法 |
|--------|------|----------|
| **输入验证** | 所有外部输入是否验证 | 检查函数参数、API 入参 |
| **SQL 注入** | 是否使用参数化查询 | 检查 SQL 拼接语句 |
| **XSS 防护** | 输出是否转义 | 检查前端渲染逻辑 |
| **认证授权** | 权限校验是否完整 | 检查接口鉴权逻辑 |
| **敏感数据** | 密码/密钥是否加密存储 | 检查配置文件、日志 |
| **依赖安全** | 第三方库是否有已知漏洞 | 检查 package.json/requirements.txt |

**安全红线**（发现即阻塞）：
- 明文存储密码
- SQL 拼接执行
- 硬编码密钥/凭证
- 无权限校验的敏感接口

---

### 2. 性能审查 (Performance) - 权重 20%

| 检查项 | 说明 | 检查方法 |
|--------|------|----------|
| **时间复杂度** | 是否存在明显性能瓶颈 | 检查嵌套循环、递归 |
| **内存管理** | 是否有内存泄漏风险 | 检查定时器、事件监听 |
| **数据库查询** | 是否存在 N+1 问题 | 检查循环内查询 |
| **缓存使用** | 热点数据是否缓存 | 检查重复计算场景 |
| **异步处理** | 耗时操作是否异步 | 检查 IO、网络请求 |

**性能红线**：
- 循环内执行数据库查询
- 大对象未释放
- 同步阻塞 IO 操作

---

### 3. 规范性审查 (Code Style) - 权重 15%

| 检查项 | 说明 |
|--------|------|
| **命名规范** | 变量、函数、类命名是否清晰一致 |
| **代码格式** | 缩进、空格、换行是否统一 |
| **注释质量** | 关键逻辑是否有注释说明 |
| **文件组织** | 文件结构是否清晰、职责单一 |
| **错误处理** | 异常捕获是否合理、错误信息是否有用 |

---

### 4. 可维护性审查 (Maintainability) - 权重 25%

| 检查项 | 说明 | 检查方法 |
|--------|------|----------|
| **单一职责** | 函数/类是否只做一件事 | 检查函数长度、职责 |
| **代码复用** | 是否存在重复代码 | 检查相似代码块 |
| **依赖管理** | 模块依赖是否清晰 | 检查 import/circular dependency |
| **可测试性** | 代码是否易于单元测试 | 检查耦合度、mock 难度 |
| **扩展性** | 是否遵循开闭原则 | 检查是否需修改原代码扩展 |

---

### 5. 测试审查 (Testing) - 权重 15%

| 检查项 | 说明 | 合格标准 |
|--------|------|----------|
| **单元测试** | 核心逻辑是否有单元测试 | 覆盖率 ≥ 80% |
| **边界测试** | 边界条件是否测试 | 空值、极值、异常值 |
| **集成测试** | 模块交互是否测试 | 关键路径覆盖 |
| **测试命名** | 测试用例命名是否清晰 | 能表达测试意图 |

## 审查流程

### Step 1: 准备阶段

```markdown
1. 确定审查范围
   - 单次提交的代码变更
   - 单个功能模块
   - 完整 PR/MR

2. 获取必要上下文
   - 需求文档（requirement.md）
   - 技术方案（tech-design.md）
   - 功能验收标准（feature_list.json）

3. 准备审查工具
   - linter 检查
   - 测试覆盖率报告
   - 安全扫描工具（如有）
```

### Step 2: 自动化检查

```bash
# 执行项目规范检查
npm run lint    # 或对应语言的工具
npm run test    # 确保测试通过
npm run coverage # 检查覆盖率
```

### Step 3: 人工审查

按照 5 个维度逐一审查，记录问题点。

### Step 4: 生成报告

输出结构化审查报告（见下方输出格式）。

## 问题分级

### 🔴 阻塞 (Blocker)
- 必须修复，否则不能合并
- 安全漏洞、严重 bug、核心功能缺失

### 🟠 严重 (Critical)
- 强烈建议修复
- 性能问题、明显的设计缺陷

### 🟡 建议 (Major)
- 推荐修复
- 代码质量问题、可维护性问题

### 🟢 建议 (Minor)
- 可选修复
- 命名优化、注释补充

## 审查技巧

### 1. 安全第一
- 先审查安全性问题
- 安全红线零容忍
- 不确定时咨询安全专家

### 2. 关注变更
- 聚焦本次变更的代码
- 不要过度审查无关代码
- 注意变更的级联影响

### 3. 提供上下文
- 说明为什么是问题
- 解释修复建议的原因
- 引用相关规范/最佳实践

### 4. 保持建设性
- 用"建议"而非"错误"
- 肯定好的设计
- 帮助而非指责

## 常见问题处理

### 1. 审查意见过多
- 按优先级排序
- 先解决阻塞/严重问题
- 建议问题可放入技术债务

### 2. 审查意见有争议
- 提供依据和规范引用
- 尊重团队技术决策
- 可升级讨论至团队评审

### 3. 时间紧张
- 优先审查安全性
- 聚焦核心功能逻辑
- 规范性可适度放宽
