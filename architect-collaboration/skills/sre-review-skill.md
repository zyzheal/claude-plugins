# SRE Review Skill - SRE 专家评审技能

本技能提供专业的站点可靠性工程 (SRE) 评审能力，覆盖 SLO 定义、故障域隔离、灾备设计、容量规划全生命周期。

## 参与阶段

| 阶段 | 评审重点 | 产出物 |
|------|----------|--------|
| **Phase 2 技术设计** | SLO 定义、故障域隔离、灾备架构 | SLO 报告、故障树、灾备方案 |
| **Phase 4 功能开发** | 降级代码、熔断实现 | 降级策略验证报告 |
| **发布评审** | 应急预案、Runbook、故障演练 | 发布批准、应急预案 |
| **上线后** | 故障复盘、SLO 达成分析 | 故障报告、改进计划 |

## 评审维度

### 1. SLO 定义 (Service Level Objective)

| 检查项 | 说明 | 合格标准 |
|--------|------|----------|
| **可用性 SLO** | 服务可用性目标 | 核心服务 ≥ 99.9%，非核心 ≥ 99% |
| **延迟 SLO** | 响应延迟目标 | p99 < 目标值（如 200ms） |
| **吞吐量 SLO** | 请求处理能力 | QPS/TPS 目标定义 |
| **错误预算** | 允许的错误率 | 基于 SLO 计算 |

**SLO 分级标准**：
```
| 服务等级 | 可用性 SLO | 年停机时间 | 适用场景 |
|----------|------------|------------|----------|
| L0 (关键) | 99.99% | < 52 分钟 | 支付、交易 |
| L1 (核心) | 99.9% | < 8.7 小时 | 用户登录、下单 |
| L2 (重要) | 99.5% | < 43 小时 | 查询、浏览 |
| L3 (一般) | 99% | < 87 小时 | 后台管理、报表 |
```

**错误预算计算**：
```
错误预算 = 100% - SLO

示例：SLO = 99.9%
错误预算 = 0.1%

如果月度请求量 = 1000 万
允许的错误请求数 = 1000 万 × 0.1% = 1 万
```

### 2. 故障域隔离 (Fault Domain Isolation)

| 检查项 | 说明 | 设计模式 |
|--------|------|----------|
| **优雅降级** | 功能降级策略 | 核心功能优先 |
| **熔断机制** | 故障服务隔离 | 熔断器模式 |
| **限流策略** | 流量过载保护 | 令牌桶/漏桶 |
| **舱壁模式** | 资源隔离 | 线程池/连接池隔离 |

**熔断器状态转换**：
```
        成功计数达到阈值
              ↓
     [CLOSED] ─────────────┐
        │                   │
  失败率超过阈值            │ 超时/半开探测成功
        ↓                   │
     [OPEN] ──────────── [HALF_OPEN]
        │                   │
        └───────────────────┘
          超时后自动转换
```

**降级策略优先级**：
```
P0: 核心功能（必须保证）
    - 用户登录
    - 支付下单

P1: 重要功能（尽量保证）
    - 商品搜索
    - 订单查询

P2: 增值功能（可降级）
    - 推荐系统
    - 个性化配置

P3: 辅助功能（优先降级）
    - 数据统计
    - 消息通知
```

### 3. 灾备设计 (Disaster Recovery)

| 检查项 | 说明 | 合格标准 |
|--------|------|----------|
| **多活/主备** | 多机房/多 Region 部署 | 核心服务双活 |
| **故障切换** | 自动/手动切换流程 | 切换时间 < RTO |
| **数据备份** | 数据冗余策略 | RPO ≤ 1 小时 |
| **容灾演练** | 定期演练计划 | 季度演练 |

**灾备级别**：
```
| 级别 | 架构 | RTO | RPO | 适用场景 |
|------|------|-----|-----|----------|
| L1 | 同城双活 | < 1 分钟 | 0 | 核心交易系统 |
| L2 | 同城主备 | < 5 分钟 | < 1 分钟 | 核心业务 |
| L3 | 异地主备 | < 30 分钟 | < 1 小时 | 重要业务 |
| L4 | 冷备 | < 4 小时 | < 4 小时 | 一般业务 |
```

### 4. 容量规划 (Capacity Planning)

| 检查项 | 说明 | 检查方法 |
|--------|------|----------|
| **资源估算** | CPU/内存/存储需求 | 基于压测/历史数据 |
| **弹性伸缩** | 自动扩缩容策略 | HPA/VPA 配置 |
| **容量预警** | 容量阈值告警 | 使用率 > 70% 告警 |
| **峰值预案** | 大促/活动预案 | 扩容方案、限流预案 |

**容量估算公式**：
```
所需实例数 = (峰值 QPS × 平均响应时间) / 单实例处理能力

示例：
- 峰值 QPS: 10000
- 平均响应时间：100ms
- 单实例处理能力：1000 QPS

所需实例数 = (10000 × 0.1) / 1000 = 1000 / 1000 = 10 实例
```

### 5. 故障应急 (Incident Response)

| 检查项 | 说明 | 合格标准 |
|--------|------|----------|
| **故障分级** | P0/P1/P2 定义清晰 | 响应时间明确 |
| **响应流程** | 发现 - 响应 - 恢复流程 | Runbook 文档化 |
| **升级机制** | 超时升级策略 | 15 分钟未响应升级 |
| **复盘机制** | 故障复盘流程 | 72 小时内完成 |

**故障分级标准**：
```
| 级别 | 影响范围 | 响应时间 | 升级时间 |
|------|----------|----------|----------|
| P0 | 核心功能完全不可用 | < 5 分钟 | 15 分钟 |
| P1 | 核心功能部分不可用 | < 15 分钟 | 30 分钟 |
| P2 | 非核心功能不可用 | < 30 分钟 | 1 小时 |
| P3 | 轻微影响 | < 1 小时 | 4 小时 |
```

## 降级策略模板

### 降级配置表

```yaml
# 降级策略配置
degradation:
  # 核心功能（不降级）
  core:
    - user.login
    - order.create
    - payment.process
  
  # 重要功能（部分降级）
  important:
    - product.search:
        strategy: cache_only  # 仅返回缓存数据
    - order.query:
        strategy: read_only   # 只读模式
  
  # 增值功能（可降级）
  value_add:
    - recommendation:
        strategy: disable     # 直接禁用
    - personalization:
        strategy: default     # 使用默认配置
  
  # 辅助功能（优先降级）
  auxiliary:
    - analytics:
        strategy: disable
    - notification:
        strategy: async       # 改为异步
```

## Runbook 模板

### 故障处理 Runbook

```markdown
# Runbook: 服务不可用应急处理

## 故障现象
- 现象描述：服务返回 5xx 错误
- 影响范围：核心功能不可用
- 故障级别：P0

## 响应流程

### Step 1: 确认故障（< 5 分钟）
1. 查看监控 Dashboard
2. 检查告警信息
3. 确认影响范围

### Step 2: 快速恢复（< 15 分钟）
1. 尝试重启服务
2. 如有配置变更，回滚配置
3. 如重启无效，切换至备用实例

### Step 3: 问题定位
1. 查看日志（ERROR 级别）
2. 检查依赖服务状态
3. 检查资源使用率

### Step 4: 修复验证
1. 验证服务恢复
2. 验证核心功能
3. 持续监控 30 分钟
```

## 常见反模式

### 1. SLO 定义反模式

```
❌ 错误：SLO 定义过高
   可用性 99.999%（年停机<5 分钟）→ 成本过高

❌ 错误：SLO 定义过低
   可用性 95%（年停机>18 天）→ 用户体验差

✅ 正确：根据业务场景定义合理 SLO
   核心交易 99.99%，查询服务 99.9%
```

### 2. 熔断设计反模式

```
❌ 错误：熔断阈值过高
   failureThreshold: 100  → 过多错误后才熔断

❌ 错误：熔断后不恢复
   没有 HALF_OPEN 状态 → 服务永久不可用

✅ 正确：合理的熔断配置
   failureThreshold: 5
   resetTimeout: 30s
   halfOpenMaxCalls: 3
```

### 3. 限流设计反模式

```
❌ 错误：限流阈值固定
   固定 1000 QPS → 无法应对流量增长

❌ 错误：限流后不降级
   拒绝请求 → 用户体验差

✅ 正确：动态限流 + 降级
   根据系统负载动态调整
   限流后返回降级数据
```
