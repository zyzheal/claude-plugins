# Architect Collaboration 快速使用指南

## 安装插件

```bash
# 安装插件
/plugins install https://github.com/zyzheal/claude-plugins.git

# 或从 marketplace 添加
/plugins marketplace add https://github.com/zyzheal/claude-plugins.git
```

## 使用方式

### 方式一：自然语言触发（推荐）

直接用自然语言描述需求，系统会自动匹配技能。

#### Phase 1: 需求分析

| 场景 | 触发语句 |
|------|----------|
| 需求分析 | "分析这个功能的需求" |
| 需求评审 | "进行需求评审" |
| 可行性评估 | "评估技术可行性" |
| 范围澄清 | "澄清功能边界和优先级" |

**示例**：
```
我需要开发一个用户管理系统，支持多角色权限控制。
请帮我分析需求，识别关键功能和约束条件。
```

#### Phase 2: 技术设计

| 场景 | 触发语句 |
|------|----------|
| 架构设计 | "设计系统架构"、"技术方案设计" |
| 数据库设计 | "设计数据库表结构"、"画 ER 图"、"设计索引策略" |
| SRE 设计 | "定义 SLO"、"设计降级策略"、"编写 Runbook" |
| 监控设计 | "设计监控方案"、"定义日志规范"、"配置告警规则" |
| 安全评审 | "做威胁建模"、"安全架构评审" |
| 合规评审 | "GDPR 合规检查"、"数据本地化方案" |

**示例**：
```
请为电商系统设计技术架构，需要考虑：
- 高并发场景（峰值 10 万 QPS）
- 数据一致性要求
- 99.99% 可用性
- 跨境业务合规
```

#### Phase 3: 任务拆解

| 场景 | 触发语句 |
|------|----------|
| 任务分解 | "拆解开发任务"、"创建任务列表" |
| 任务评审 | "评审任务规划和依赖关系" |
| 测试规划 | "规划测试任务和覆盖率要求" |

**示例**：
```
根据技术设计方案，帮我拆解开发任务。
任务粒度控制在 2 人天以内，明确依赖关系和优先级。
```

#### Phase 4: 功能开发

| 场景 | 触发语句 |
|------|----------|
| 功能实现 | "实现用户登录功能"、"编写代码" |
| TDD 开发 | "用 TDD 方式开发支付功能" |
| 进度跟踪 | "更新开发进度" |

**示例**：
```
开始实现用户认证模块，按照 TDD 流程：
1. 先写测试用例
2. 实现最小可用代码
3. 重构优化
```

#### 评审阶段

| 场景 | 触发语句 |
|------|----------|
| 代码审查 | "审查这段代码"、"代码评审" |
| 安全评审 | "安全检查"、"漏洞扫描" |
| 数据库评审 | "评审数据库设计"、"SQL 性能分析" |
| SRE 评审 | "评审 SLO 定义"、"检查降级策略" |
| 监控评审 | "评审监控方案"、"检查告警规则" |
| 合规评审 | "合规检查"、"隐私保护评审" |

**示例**：
```
请审查以下代码的安全性：
- 认证授权逻辑
- 输入验证
- 敏感数据处理
- OWASP Top 10 防护
```

## 完整工作流示例

### 示例：开发用户管理系统

```
Step 1: 需求分析
> "我需要开发一个多角色用户管理系统，请分析需求"

Step 2: 技术设计
> "基于需求分析，设计系统架构和数据库表结构"
> "定义 SLO 和监控指标"
> "做威胁建模和安全设计"

Step 3: 任务拆解
> "根据技术设计，拆解开发任务"

Step 4: 功能开发
> "开始实现用户认证模块，使用 TDD 方式"

Step 5: 代码审查
> "审查已完成的代码"

Step 6: 发布评审
> "进行发布前安全评审和合规检查"
```

## 输出产物

每个阶段都会生成标准化的文档：

| 阶段 | 产物 | 位置 |
|------|------|------|
| Phase 1 | 需求文档 (PRD)、风险矩阵 | `docs/requirement.md` |
| Phase 2 | 技术设计文档、架构图 | `docs/tech-design.md` |
| Phase 3 | 开发任务列表 | `docs/开发任务.md` |
| Phase 4 | 代码、测试、覆盖率报告 | `src/` |
| 评审 | 审查报告 | `docs/reviews/` |

## 最佳实践

### 1. 按顺序执行
```
需求分析 → 技术设计 → 任务拆解 → 功能开发 → 评审
```

### 2. 提供充分上下文
```
❌ "设计数据库"
✅ "为电商订单系统设计数据库，预计日订单量 100 万，需要支持：
    - 订单查询（按用户、时间、状态）
    - 订单金额统计
    - 数据保留 3 年"
```

### 3. 明确约束条件
```
"设计高可用架构，要求：
- 可用性 99.99%
- RTO < 1 分钟
- RPO = 0
- 支持异地多活"
```

### 4. 迭代完善
```
"基于上次的设计，补充缓存层方案"
"根据评审意见，优化索引策略"
```

## 常见问题

### Q: 技能没有触发怎么办？
A: 尝试更明确的触发语句，或直接使用技能名称。

### Q: 如何查看已生成的文档？
A: 所有文档默认生成在项目 `docs/` 目录下。

### Q: 如何切换阶段？
A: 直接说明需求，如"我们进入 Phase 3 任务拆解阶段"。

### Q: 如何评审特定代码？
A: 提供代码路径或内容，然后说"请审查这段代码"。
